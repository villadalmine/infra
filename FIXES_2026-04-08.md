## Root Causes

### Bug 1 — `kubernetes-mcp-server` exits with code 0

**Cause**: The sidecar command at `hermes-static-mcp.yaml.j2:141` called:
```sh
exec /app/kubernetes-mcp-server
```

Two problems:
1. Binary is at `/usr/local/bin/kubernetes-mcp-server`, not `/app/`
2. No `--port 8080` → binary runs in stdio mode, reads EOF, exits 0

**Evidence**:
- `kubectl describe pod`: Exit Code 0, restarts every ~1s
- `kubectl logs`: "read error: EOF"
- Dockerfile: `ENTRYPOINT ["/usr/local/bin/kubernetes-mcp-server"]`

### Bug 2 — Hermes: `No messaging platforms enabled`

**Cause**: The ConfigMap `hermes-gateway-config` existed in cluster with correct `gateway.json`, but:
- `tasks/main.yml` had no task to apply it
- `hermes-static-mcp.yaml.j2` had no volume/volumeMount for it

**Evidence**:
- `kubectl logs hermes-agent`: "WARNING gateway.run: No messaging platforms enabled."
- ConfigMap exists with correct Telegram config
- Template exists (`hermes-gateway-configmap.yaml.j2`) but not applied

### Bug 3 — Hermes: `401 - Missing Authentication header`

**Cause**: Hermes reads `.env` file first, then falls back to environment variables. Without a `.env` file, it doesn't send the `Authorization` header even though `OPENAI_API_KEY` is in the environment.

**Evidence**:
- Logs: "No .env file found. Using system environment variables."
- Logs: "Error code: 401 - {'error': {'message': 'Missing Authentication header', 'code': 401}}"
- LiteLLM works fine when tested directly with `curl -H "Authorization: Bearer sk-hermes-internal"`

### Bug 4 — kagent `ModelConfig` has empty `secretHash`

**Cause**: The Ansible passed `apiKey: sk-hermes-internal` (inline) in Helm values, but the kagent chart supports `apiKeySecretRef` (recommended pattern). Using inline `apiKey` worked in some versions but left `secretHash` empty.

**Evidence**:
- Live ModelConfig: `secretHash: e3b0c44...` (SHA256 of empty string)
- Manual patch to add `apiKeySecret: kagent-openai` fixed it immediately
- Chart values.yaml shows `apiKeySecretRef` and `apiKeySecretKey` as the correct fields

### Fix 4 — `hermes-static-mcp.yaml.j2`: create `.env` file via initContainer

**Required**: Hermes OpenAI client does NOT configure Authorization header correctly when using environment variables alone. A `.env` file is mandatory for reliable authentication.

**File**: `roles/install-hermes-agent/templates/hermes-static-mcp.yaml.j2`

**Add `initContainers` section before `containers`**:
```yaml
      initContainers:
        - name: create-env-file
          image: busybox:latest
          command: ["sh", "-c"]
          args:
            - |
              cat <<EOF > /opt/data/.env
              OPENAI_API_KEY={{ hermes_openai_api_key }}
              OPENAI_API_BASE={{ hermes_openai_api_base }}
              EOF
              chmod 644 /opt/data/.env
          volumeMounts:
            - name: hermes-data
              mountPath: /opt/data
```

**Evidence from testing**:
- Environment variables present: `OPENAI_API_KEY=sk-hermes-internal`, `OPENAI_API_BASE=http://litellm-proxy...`
- Hermes logs: `No .env file found. Using system environment variables.`
- Immediately followed by: `Error code: 401 - {'error': {'message': 'Missing Authentication header'}}`
- LiteLLM tested independently with `curl` + `Authorization: Bearer sk-hermes-internal` → works perfectly
- Conclusion: Hermes OpenAI client fails to set auth header when reading from env vars

### Fix 5 — `hermes-static-mcp.yaml.j2`: remove unsupported `HERMES_INFERENCE_PROVIDER`

**File**: `roles/install-hermes-agent/templates/hermes-static-mcp.yaml.j2`

**Remove** the env var:
```diff
-            - name: HERMES_INFERENCE_PROVIDER
-              value: {{ hermes_inference_provider }}
```

Hermes 0.7.0 doesn't use this variable. It reads provider config from `.env` file.

### Fix 6 — `install-kagent/tasks/main.yml`: use `apiKeySecretRef` instead of inline `apiKey`

**File**: `roles/install-kagent/tasks/main.yml`

**Add before Helm install** (creates the secret first):
```yaml
- name: Create kagent OpenAI API key secret
  kubernetes.core.k8s:
    state: present
    definition:
      apiVersion: v1
      kind: Secret
      metadata:
        name: kagent-openai
        namespace: "{{ kagent_namespace }}"
      type: Opaque
      stringData:
        OPENAI_API_KEY: "{{ kagent_llm_api_key }}"
```

**Change Helm values** (lines 95-102):
```diff
       providers:
         default: openAI
         openAI:
           provider: OpenAI
           model: "{{ kagent_llm_model }}"
-          apiKey: "{{ kagent_llm_api_key }}"
+          apiKeySecretRef: kagent-openai
+          apiKeySecretKey: OPENAI_API_KEY
           config:
             baseUrl: "{{ kagent_llm_base_url }}"
```

This is the **correct pattern** per the kagent Helm chart. No manual patches needed.

## Security Architecture — Zero Direct API Access

**Design principle**: No agent has direct access to OpenRouter or any external LLM provider. All traffic flows through LiteLLM.

| Agent | API Key | Base URL | Real OpenRouter Key |
|-------|---------|----------|---------------------|
| **Hermes** | `sk-hermes-internal` (LiteLLM) | `litellm-proxy.ai:4000` | ❌ No access |
| **Holmes** | `sk-hermes-internal` (LiteLLM) | `litellm-proxy.ai:4000` | ❌ No access |
| **kagent** | `sk-hermes-internal` (LiteLLM) | `litellm-proxy.ai:4000` | ❌ No access |
| **OpenClaw** | `sk-hermes-internal` (LiteLLM) | `litellm-proxy.ai:4000` | ❌ No access |
| **LiteLLM** | N/A (proxy layer) | N/A | ✅ Has all 4 real keys |

**Fallback chains** (managed by LiteLLM):
- `hermes-qwen → free2 → cheap`
- `holmes-llama → holmes-free2 → holmes-cheap`
- `openclaw-gemini → gemini-free2 → cheap`
- `kagent-orchestrator → cheap`

**Benefits**:
1. **Single point of control**: all external API calls visible in LiteLLM metrics
2. **Cost tracking**: Grafana dashboard shows spend per agent/model
3. **Security**: agents can't bypass rate limits or fallback rules
4. **Flexibility**: change models/providers without redeploying agents

**Implementation**:
- All agents use `OPENAI_API_BASE` pointing to LiteLLM
- All agents use `OPENAI_API_KEY=sk-hermes-internal` (shared internal key)
- LiteLLM uses separate OpenRouter keys per agent for cost attribution
- No agent has environment variables or config files with real OpenRouter keys
